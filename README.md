# BrainGame Project

`BrainGame Project` — это прошивка для `Arduino Uno`, которая управляет системой кнопок для интеллектуальных игр, квизов и любых раундов, где важно быстро определить, какая команда нажала первой.

Если объяснять совсем просто, устройство делает следующее:
- есть одна мастер-кнопка, которая запускает и останавливает раунд;
- есть четыре игровые кнопки для четырёх команд;
- есть четыре индикатора команд;
- есть один общий предупреждающий индикатор;
- после старта система ждёт первую нажатую команду, фиксирует её и блокирует остальные;
- если никто не нажал кнопку вовремя, раунд сам завершается по таймеру.

Этот репозиторий нужен не только для хранения прошивки, но и для того, чтобы её можно было:
- собирать одинаково у любого разработчика;
- автоматически проверять тестами;
- автоматически собирать в GitHub Actions;
- отдавать готовый `firmware.hex` как артефакт после сборки.

## Что именно делает устройство

У прошивки есть три состояния.

### `STOP`

Это режим ожидания.

Что происходит в этом режиме:
- раунд не запущен;
- таймер не идёт;
- если нажать любую игровую кнопку, соответствующий индикатор просто мигает;
- нажатие игровой кнопки не считается ответом и не запускает раунд.

### `RUNNING`

Это активный раунд.

Что происходит в этом режиме:
- мастер-кнопка уже нажата, раунд стартовал;
- начинает идти таймер ответа;
- система ждёт первую команду, которая нажмёт свою кнопку;
- если до конца времени никто не ответил, раунд завершится автоматически.

### `LOCKED_BY_TEAM`

Это режим после первого ответа.

Что происходит в этом режиме:
- первая нажатая команда уже зафиксирована;
- загорается её индикатор;
- остальные команды больше не могут “перебить” ответ;
- таймер больше не продолжает выбирать победителя;
- выход из этого состояния происходит через мастер-кнопку, которая возвращает систему в `STOP`.

## Логика раунда

Текущая логика такая:

1. Система включается в состоянии `STOP`.
2. Ведущий нажимает мастер-кнопку.
3. Система переходит в `RUNNING`.
4. Таймер начинает отсчёт.
5. Если одна из команд нажала первой, система:
   фиксирует именно эту команду;
   включает её индикатор;
   игнорирует остальные игровые кнопки до конца раунда.
6. Если время дошло до предупреждающей точки, включается общий warning-индикатор.
7. Если время вышло и никто не ответил, система автоматически возвращается в `STOP`.
8. Ведущий в любой момент активного раунда может нажать мастер-кнопку и вручную остановить раунд.

## Текущие настройки

По умолчанию в проекте используются такие параметры:

- длительность раунда: `60 секунд`
- предупреждение: `40 секунда`
- число игровых команд: `4`

Используемые пины:

- мастер-кнопка: `8`
- warning-индикатор: `7`
- кнопки команд: `9`, `10`, `11`, `12`
- индикаторы команд: `2`, `3`, `4`, `5`

## Для кого эта структура проекта

Проект специально приведён к структуре, которую легче поддерживать, чем один большой Arduino-скетч.

Это полезно, если нужно:
- править логику таймера;
- менять правила раунда;
- тестировать проект без платы;
- не ломать случайно рабочее поведение при следующем изменении.

## Структура проекта

### `src/main.cpp`

Это тонкая Arduino-обвязка.

Она отвечает только за:
- чтение физических пинов;
- перевод состояний кнопок в входные данные логики;
- применение результатов логики к `digitalWrite`.

Здесь не должна жить сложная игровая логика.

### `lib/GameCore/src/GameController.h`
### `lib/GameCore/src/GameController.cpp`

Это основная логика проекта.

Именно здесь находится:
- state machine;
- правила переходов между `STOP`, `RUNNING`, `LOCKED_BY_TEAM`;
- таймер;
- warning;
- блокировка повторных нажатий;
- логика мигания индикаторов в режиме `STOP`.

Эта часть специально написана без зависимости от `Arduino.h`, чтобы её можно было тестировать как обычный C++-код.

### `test/test_game_controller/test_main.cpp`

Это автоматические unit-тесты.

Они проверяют игровую логику без реальной платы.

### `tests`

Это ручной чек-лист для проверки на реальном железе.

Он нужен потому, что автотесты проверяют логику, но не могут проверить:
- реальную проводку;
- физические кнопки;
- возможный дребезг конкретных компонентов;
- реальную индикацию на собранном устройстве.

### `platformio.ini`

Это файл конфигурации `PlatformIO`.

В проекте настроены два окружения:
- `uno` — сборка реальной прошивки для `Arduino Uno`;
- `native` — запуск unit-тестов на обычном компьютере.

## Что уже автоматизировано

В проект добавлен GitHub Actions workflow:
- он запускается на `push`, `pull_request` и вручную;
- прогоняет unit-тесты;
- собирает прошивку для `Arduino Uno`;
- сохраняет готовый `firmware.hex` как артефакт.

Это и есть практическая CI/CD-схема для этого проекта:
- `CI` — автоматическая проверка логики и сборки;
- `CD` — автоматическая сборка и публикация готового файла прошивки как артефакта pipeline.

Workflow лежит в:
- `.github/workflows/ci-cd.yml`

## Что нужно установить с нуля

Если человек впервые открывает этот проект, ему нужен только Python и `PlatformIO`.

### 1. Убедиться, что есть Python

Проверка:

```bash
python3 --version
```

### 2. Установить PlatformIO

```bash
python3 -m pip install --user platformio
```

### 3. Проверить, что PlatformIO доступен

```bash
python3 -m platformio --version
```

Такой вызов надёжнее, чем просто `pio`, потому что он не зависит от того, добавился ли путь к бинарнику в `PATH`.

## Как собрать проект

Сборка прошивки для `Arduino Uno`:

```bash
python3 -m platformio run -e uno
```

После успешной сборки получится файл:

```text
.pio/build/uno/firmware.hex
```

Это и есть готовая прошивка.

## Как прогнать автотесты

Для проверки логики без платы:

```bash
python3 -m platformio test -e native
```

Что проверяют эти тесты:
- старт раунда по мастер-кнопке;
- ручной останов раунда;
- фиксацию первой нажатой команды;
- игнорирование остальных кнопок после фиксации;
- warning на 40-й секунде;
- автосброс на 60-й секунде;
- поведение кнопок в `STOP`;
- корректный переход между несколькими раундами;
- защиту от повторного срабатывания при удержании кнопки;
- завершение мигания в `STOP`.

## Рекомендуемая локальная последовательность

Если вы что-то поменяли в коде, нормальный цикл такой:

```bash
python3 -m platformio test -e native
python3 -m platformio run -e uno
```

Сначала логика, потом прошивка.

## Как проверить на реальной плате

После сборки и загрузки прошивки в `Arduino Uno` вручную прогоните сценарии из файла `tests`.

Минимальный набор проверок:

1. Нажать мастер-кнопку и проверить старт раунда.
2. Нажать кнопку команды 1, затем попытаться нажать кнопку другой команды.
3. Убедиться, что засчитывается только первая команда.
4. Проверить warning на 40-й секунде.
5. Проверить автоматический возврат в `STOP` на 60-й секунде, если никто не ответил.
6. Проверить ручной stop мастер-кнопкой.
7. Проверить, что в `STOP` игровые кнопки не запускают таймер, а только мигают.
8. Проверить несколько раундов подряд.

## Где менять настройки

Если нужно поменять пины, это делается в:
- `src/main.cpp`

Если нужно поменять правила логики, это делается в:
- `lib/GameCore/src/GameController.h`
- `lib/GameCore/src/GameController.cpp`

Если нужно поменять длительность раунда, warning или параметры мигания, это делается через `GameConfig`.

## Что в проект пока не входит

На текущем этапе проект не делает следующее:
- не работает с дисплеем;
- не имеет меню настройки времени с кнопок;
- не проверяет аппаратные проблемы автоматически;
- не загружает прошивку на плату из CI.

Последний пункт нормален: GitHub Actions может собрать и отдать прошивку, но физическую загрузку в реальную `Arduino Uno` всё равно делает человек локально.
